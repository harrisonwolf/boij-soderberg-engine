# boij-soderberg-engine

C++ tooling and datasets for searching degree sequences in Boij-Soderberg style experiments, with a focus on finding and post-processing sequences that violate BEH/LLBC checks derived from pure Betti tables.

## What This Repository Is For

This repository is a research workflow engine for:

- generating candidate degree sequences in fixed codimension,
- computing pure Betti sequences from those degree sequences,
- testing conjectural constraints (BEH and LLBC),
- collecting candidate violators ("bad ones"),
- parsing and deduplicating large batch outputs.

It is intentionally CLI-first and scriptable, with historical data and research snapshots kept in-repo for reproducibility.

## What It Is Not

- Not a polished end-user package.
- Not a general-purpose Boij-Soderberg library API.
- Not guaranteed to be optimized for very large runs without parameter tuning.

## Core Concepts in This Codebase

- **Degree sequence**: modeled as `{0,d1,d2,...,dc}` with strictly increasing entries.
- **Codimension (`c`)**: inferred as `sequence_length - 1`.
- **Pure Betti sequence**: computed from a degree sequence (`pure_betti` in `src/seq_funcs.cc`).
- **Checks**:
  - `test_BEH(...)` checks entrywise lower bounds against binomial coefficients.
  - `test_LLBC(...)` checks a lower bound on total Betti sum.
- **"Bad ones"**: sequences failing one or both checks.
- **Rinsing/deduplication**:
  - `gcd_rinse(...)` is a quick filter.
  - `rinse_seqs(...)` removes constant-multiple duplicates more thoroughly.

## Repository Layout

- `apps/`: CLI executables (`main` programs).
- `src/`: shared implementation (`seq_funcs`, `test_funcs`, `binom`).
- `include/`: headers used by applications.
- `scripts/`: batch helper scripts for generation/merge/parse/rinse workflows.
- `data/`:
  - `raw/`: generated and manually collected raw outputs.
  - `processed/`: parsed/rinsed outputs.
  - `samples/`: small files for testing commands quickly.
  - `archive/`: historical outputs.
- `docs/`: migration and legacy notes.
- `research/`: archived snapshots, exploratory artifacts, and backups.
- `build/`: generated by `make` (`build/bin`, `build/obj`).

## Requirements

- `g++` (or compatible C++ compiler)
- `make`

If your compiler defaults are older, build with C++17 explicitly:

```bash
make CXXFLAGS="-O2 -Wall -Wextra -std=c++17"
```

## Build

From repository root:

```bash
make
```

Default binaries produced in `build/bin/`:

- `bad_one_generator`
- `parse_huge_output`
- `remove_duplicates`
- `test_program`
- `tell_which_violations`
- `find_big_ones`

Build extra tools:

```bash
make L_finder quick_run find_172 foo
```

Clean build output:

```bash
make clean
```

## Quick Start

### 1) Generate a small "bad ones" file

```bash
touch /tmp/bad_ones_demo.txt
./build/bin/bad_one_generator /tmp/bad_ones_demo.txt 3 30 1
```

Arguments are:

```text
bad_one_generator [outputfile] [codimension c] [max degree d] [optional: lowbound l]
```

### 2) Parse noisy merged output into clean sequence lines

```bash
touch /tmp/parsed_demo.txt
./build/bin/parse_huge_output data/samples/burntest.txt /tmp/parsed_demo.txt
```

### 3) Remove constant-multiple duplicates

```bash
touch /tmp/unique_demo.txt
./build/bin/remove_duplicates /tmp/parsed_demo.txt /tmp/unique_demo.txt
```

### 4) Show which checks each sequence violates

```bash
./build/bin/tell_which_violations /tmp/unique_demo.txt
```

## Executable Reference

### `bad_one_generator`

Usage:

```text
bad_one_generator [outputfile] [codimension c] [max degree d] [optional: lowbound l]
```

What it does:

- Generates degree sequences via `gen_deg_seqs(c,d,l)`.
- Tests each sequence against BEH/LLBC through `test_conjs(...)`.
- Writes all violators and a `gcd_rinsed` subset to `outputfile`.

Notes:

- The output file **must already exist** (`touch file.txt` first).
- For large `(c,d)` this can become expensive quickly.

### `parse_huge_output`

Usage:

```text
parse_huge_output [input file] [output file]
```

What it does:

- Scans merged/noisy logs.
- Extracts brace-form sequences.
- Writes one sequence per line to output.

Notes:

- Output file must already exist.
- Designed for post-processing merged burn outputs.

### `remove_duplicates`

Usage:

```text
remove_duplicates [input_file.txt] [output_file.txt]
```

What it does:

- Reads sequences from input.
- Removes sequences that are constant multiples of earlier ones.
- Writes rinsed list to output.

Notes:

- Output file must already exist.
- Best with compact sequence format like `{0,1,2,3}` (no spaces).

### `tell_which_violations`

Usage:

```text
tell_which_violations [input filename]
```

What it does:

- For each sequence, prints which `B_i` values fail BEH and whether LLBC fails.
- Prints `(passed both)` when neither check fails.

Notes:

- Always pass an input filename argument.
- Current implementation may also print internal debug lines to stderr.

### `test_program`

Usage:

```text
test_program
```

What it does:

- Interactive driver for development/testing functions:
  - `gen_deg_seqs`, `pure_betti`, `test_conjs_v2`, `gen_deg_seqs_v2`, `is_degen`, `pi`, `calc_L`.

### `find_big_ones`

Usage:

```text
find_big_ones
```

What it does:

- Runs a built-in scan (`gen_deg_seqs(3,100)`).
- Prints sequences whose pure Betti entries exceed an internal threshold.

### Extra/Experimental executables

- `L_finder`: exploratory search for self-dual patterns with specific `L`.
- `quick_run`: ad-hoc targeted test loop.
- `find_172`: scaffold for a specific search target.
- `foo`: empty stub.

These are primarily research scratchpads, not stable interfaces.

## Scripted Batch Workflows

Scripts are in `scripts/` and expect binaries in `build/bin/`.

### `scripts/run_tests.sh`

- Runs `bad_one_generator` repeatedly over degree windows.
- Writes per-window files in `data/raw/burnfiles/c*/`.
- Appends each window output into `huge_output.txt`.
- Does **not** truncate an existing `huge_output.txt` first.

Important defaults currently hardcoded in script:

- `c=8`
- `increment=1`
- loop `n=29..500`

Edit these before running long jobs.

Run:

```bash
bash scripts/run_tests.sh
```

### `scripts/merge_outputs.sh`

- Concatenates burn chunks into a codimension-specific `huge_output.txt`.

Defaults currently hardcoded:

- `c=6`
- `increment=2`

Run:

```bash
bash scripts/merge_outputs.sh
```

### `scripts/parse_and_rinse.sh`

- Parses `data/raw/burnfiles/c*/huge_output.txt` into:
  - `huge_output_parsed.txt`
  - `huge_output_unique.txt`

Current default codimension in script:

- `c=8`

Run:

```bash
bash scripts/parse_and_rinse.sh
```

## Data Format and Parsing Expectations

Preferred per-line sequence format:

```text
{0,1,2,3}
```

Parser behavior:

- It scans for `{...}` patterns and extracts comma-separated integers.
- Header text/noise is tolerated.
- Whitespace-heavy sequence forms (for example `{0, 1, 2, 3}`) are less reliable for `remove_duplicates`; prefer compact format with no spaces.

## Typical End-to-End Pipeline

1. Build tools with `make`.
2. Generate chunk outputs with `bad_one_generator` (or `scripts/run_tests.sh`).
3. Merge chunks into `huge_output.txt` (if chunked).
4. Parse with `parse_huge_output`.
5. Rinse duplicates with `remove_duplicates`.
6. Inspect violations with `tell_which_violations`.
